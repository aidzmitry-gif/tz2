# Дашборд аналитики продаж

Аналитическая панель для продаж: фильтруемая таблица заказов с серверной пагинацией,
графики (выручка по периодам, воронка конверсии, распределение по каналам), сравнение
«текущий vs предыдущий» период равной длины и экспорт отфильтрованной выборки в Excel/CSV.

Реализация ТЗ №2. Акцент — на правильной архитектуре отчётности: вся тяжёлая агрегация
выполняется в БД, на клиент уходят только сжатые результаты.

## Возможности

- **Таблица заказов** — серверная пагинация (`range()`) и сортировка по колонкам (ID, дата,
  канал, статус, сумма). На клиент отдаётся только текущая страница из 10 000+ строк.
- **Фильтры** — период (от/до), статус, канал, сегмент. Полностью синхронизированы с URL
  (query params): состояние шарится ссылкой, кнопка «Назад» работает корректно. Ввод дат
  дебаунсится (450 мс), чтобы не спамить запросами.
- **Графики (Recharts)** — выручка по дням/неделям/месяцам (area + линия числа заказов),
  воронка конверсии по статусам, доля каналов в выручке (донат).
- **Сравнение периодов** — KPI (выручка, заказы, средний чек, конверсия в оплату) с дельтой
  к предыдущему периоду равной длины. Относительный % для денег и количеств, процентные
  пункты (п.п.) — для конверсии.
- **Экспорт** — выгрузка текущей выборки в `.xlsx` (даты — датами, суммы — числами, с
  числовыми форматами) и `.csv` (UTF-8 с BOM, корректно открывается в Excel).
- **Кэш** — тяжёлые RPC обёрнуты в `unstable_cache` (revalidate 60 c), ответы отдаются с
  `Cache-Control`, на клиенте SWR с `keepPreviousData` (данные не «мигают» при смене фильтра).

## Стек

- **Next.js 14** (App Router) + **TypeScript**
- **Supabase** (Postgres) — данные и RPC-функции агрегации
- **Recharts** — графики
- **SheetJS (xlsx)** — экспорт в Excel/CSV
- **SWR** — клиентский кэш и ревалидация
- **Tailwind CSS** — стили

## Быстрый старт

### 1. Установка зависимостей

```bash
npm install
```

### 2. Переменные окружения

Скопируйте `.env.example` в `.env.local` и подставьте URL и ключи из проекта Supabase
(Project Settings → API):

```bash
cp .env.example .env.local
```

```
NEXT_PUBLIC_SUPABASE_URL=https://your-project.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=your-anon-key
```

Anon-ключа достаточно: в миграции включён RLS с публичными политиками на `SELECT`.

### 3. Схема и данные

Примените миграции и сид. Самый простой путь — через **SQL Editor** в дашборде Supabase:
выполните по очереди содержимое файлов

1. `supabase/migrations/0001_init.sql`
2. `supabase/migrations/0002_functions.sql`
3. `supabase/seed.sql`

Либо через Supabase CLI / `psql`:

```bash
# через psql (подставьте строку подключения проекта)
psql "$DATABASE_URL" -f supabase/migrations/0001_init.sql
psql "$DATABASE_URL" -f supabase/migrations/0002_functions.sql
psql "$DATABASE_URL" -f supabase/seed.sql
```

### 4. Запуск

```bash
npm run dev
```

Откройте http://localhost:3000.

## RPC-функции

Вся агрегация — в Postgres (`date_trunc` + `group by`), сырые строки на клиент не тянутся.

- `revenue_by_period(p_from, p_to, p_granularity, p_status, p_channel, p_segment)` — выручка и
  число заказов по бакетам (день/неделя/месяц).
- `conversion_funnel(p_from, p_to, p_channel, p_segment)` — накопительная воронка по статусам
  (new → processing → paid → shipped → delivered); cancelled/refunded в воронку не входят.
- `channel_distribution(p_from, p_to, p_status, p_segment)` — выручка и заказы по каналам.
- `period_comparison(p_from, p_to, p_status, p_channel, p_segment)` — метрики текущего и
  предыдущего (равной длины) периода одним запросом, возвращает JSON.

## Технические заметки

- **Границы периода.** `to` трактуется как включительный день: верхняя граница — начало
  следующего дня (`< toTs`), единообразно в таблице и всех RPC. Границы дня считаются в UTC —
  для демо это допущение; под конкретную таймзону можно домножить на смещение в RPC.
- **Сравнение периодов.** Предыдущий период = `[from − (to − from), from)`, ровно той же
  длины, что и текущий. Проверяется вручную на контрольных датах.
- **Пагинация.** Сейчас offset-пагинация через `range()` — её достаточно для условий ТЗ.
  Для совсем больших объёмов и «бесконечной» прокрутки можно перейти на keyset-пагинацию
  (по `created_at`, `id`); индекс под это уже заложен.
- **Альтернатива RPC.** При росте нагрузки агрегаты по выручке можно вынести в materialized
  view с периодическим refresh — интерфейс ответа менять не придётся.
- **Кэш.** Слой `unstable_cache` ключуется по сериализованным фильтрам, поэтому разные
  выборки не вытесняют друг друга; TTL — 60 секунд.
